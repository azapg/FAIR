FAIR chat — conversation taxonomy

A structured spec for conversation trajectories, message anatomy, and UI event types across all FAIR agent roles.

What this is

FAIR's chat UI is not a chatbot — it is a long-running agent orchestration surface. Conversations can span minutes to hours, emit dozens of artifacts, spawn sub-agents, request human approval, and resume across sessions. This taxonomy defines every state, event, and structural unit the UI must be capable of rendering.

Seven message categories

Human turn
Text, file uploads, voice, in-line references to prior artifacts or grades.
Agent turn
One or many content blocks: text, CoT, tool calls, tool results, final answer.
Tool event
Structured call + structured result. Can nest (sub-agent launches another agent).
Artifact
Chart, dashboard, PDF, slide deck, code file, video, simulation output, grading report.
Interrupt
Agent pauses and asks the human for a decision, approval, or clarification.
Status pulse
Lightweight heartbeat streamed during long work. Keeps the UI alive without spamming.
Error / recovery
Tool failure, partial output, retry state, or agent-level exception with cause.

Conversation lifecycle states

// A conversation moves through these states, sometimes non-linearly
IDLE → STREAMING → TOOL_RUNNING → INTERRUPTED → STREAMING → COMPLETE
                        ↳ ERROR → (retry | abandon)
                        ↳ SUB_AGENT_RUNNING → STREAMING
// INTERRUPTED pauses streaming until human resolves the interrupt
// COMPLETE can have pending artifacts still rendering client-side

Key design constraints

◆Agents work for minutes or hours. The UI must stay alive with status pulses and never time out optimistically. Always show elapsed time and last activity.
◆A single agent turn may emit 0–N artifacts. Artifacts appear inline as they are produced, not at the end. Each has an independent render state.
◆Chain-of-thought (CoT) is auto-collapsed when the final message arrives. Users can re-expand it. Never discard CoT — it is research data in FAIR's context.
◆Tool calls and results can be nested: a grading agent can launch a sub-agent per student submission. The UI must handle recursive tool/result blocks to arbitrary depth.
◆Role-awareness: the UI renders differently for student, professor, and researcher contexts. Same conversation model; different default collapsed states and action affordances.

Role-differentiated UI behavior

Same underlying conversation model; different default presentation, affordances, and suggested actions per role.

Student view

CoT visible but minimal by default. Show "thinking" indicator only; expand on tap. Students should focus on the answer, not the reasoning machinery.
Artifacts emphasize learning. Interactive simulations, step-by-step worked solutions, visualizations. No grading reports, no batch views.
No approval interrupts. Students don't make grading decisions. Interrupt blocks are filtered from student view or replaced with "waiting for instructor" placeholder.
Practice mode persistence. Conversation stores which problems were attempted, errors made, and progress so adaptive problem selection works across sessions.

Professor view

Tool calls and results visible collapsed. Professors want to trust but verify. Tool calls expand on click with full input/output. Sub-agent chains are summarized as "32 sub-tasks completed".
Interrupts are primary UI. Interrupt blocks are prominent, scroll into view automatically, and block the input field until resolved.
Dashboard artifacts are first-class. Grade distribution charts, performance tables, and KC mastery views are full-width and pin to the artifact panel.
Inline override affordance. Professors can override any AI-assigned grade directly from the artifact without re-prompting. Override triggers a record in the grading audit log.

Researcher view

Full CoT always expanded. Researchers want to see the agent's reasoning in detail. CoT is never auto-collapsed; it's treated as primary data.
Raw tool outputs available. Every tool call has a "show raw JSON" toggle. Researchers can copy tool inputs/outputs directly for their own analysis pipelines.
Session export. Full conversation — messages, tool calls, CoT, artifacts — exportable as structured JSON or Markdown for research documentation.
Analysis artifacts are default full-screen. Clustering maps, embedding projections, and diagnostic visualizations open in a full-width canvas panel, not an inline bubble.

Server-sent event stream contract

FAIR agents stream events to the client via SSE. Every event has a type field. The UI renders blocks incrementally as events arrive.

Event types (aligns with AI SDK)

text_delta
string
Incremental token. Appended to current prose block. Differentiates CoT vs final by content_block_type.
tool_use_start
object
Opens a tool call block. Fields: tool_name, tool_input (streamed), call_id. Render as collapsible row.
tool_result
object
Closes a tool call. Fields: call_id, content (raw), is_error. Appended to the matching tool_use_start block.
artifact_start
object
Opens an artifact block. Fields: artifact_id, type (html|csv|pdf|mp4|pptx|image), title, render_inline.
artifact_chunk
bytes
Binary chunk for artifact content (used for large files). Accumulated client-side until artifact_end.
artifact_end
object
Closes artifact. Fields: artifact_id, size_bytes, download_url, render_complete.
status_pulse
object
Heartbeat during long work. Fields: message, progress (0–1 or null), elapsed_ms. Replaces prior pulse in UI (not a new block).
interrupt
object
Pauses stream until resolved. Fields: interrupt_id, title, body, options[] (label + value pairs). UI blocks input until user picks an option and sends interrupt_response.
subagent_start
object
A nested agent has been launched. Fields: agent_id, purpose, parent_tool_call_id. Shown as indented sub-block.
subagent_done
object
Nested agent completed. Summarized as "N sub-tasks completed" in collapsed state.
error
object
Fields: code, message, recoverable, retry_after_ms. Renders error block with retry affordance if recoverable.
message_stop
object
Turn complete. Fields: stop_reason (end_turn | tool_use | interrupt | error), usage. Triggers CoT auto-collapse in student/professor views.

Artifact type → render strategy

html
Render in sandboxed iframe inline. Width: 100%. Height: auto from content.
csv / xlsx
Preview as table (first 50 rows). Full download link + open in spreadsheet.
pdf
Inline PDF viewer (first page preview + scroll). Full download.
mp4
Inline video player with autoplay=false. Thumbnail shown until play.
pptx
Static slide thumbnails (1 per slide). Download link. No live render client-side.
image
Inline img with lightbox on click. Alt text required from agent.

AI SDK / Python SDK alignment notes

◆text_delta maps to AI SDK's TextStreamPart. Use content_block_type: "thinking" vs "text" to differentiate CoT from final prose.
◆tool_use_start / tool_result map to ToolCallStreamPart / ToolResultStreamPart. FAIR adds sub_agent flag for nested launches.
◆artifact_* events are FAIR-specific extensions — not in base AI SDK. They wrap a tool result that produces a renderable file, tagged with a FAIR artifact envelope.
◆status_pulse is a FAIR extension. Emitted by the orchestrator every 5–15s during long tool runs. Replaces rather than appends — client keeps only the latest pulse per turn.
◆interrupt is a FAIR extension. When received, stream is paused server-side. Client must POST /conversation/{id}/interrupt/{interrupt_id}/resolve with the user's choice to resume.
